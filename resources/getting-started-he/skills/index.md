# כישורים ב-Cabinet

כישורים הם חבילות הוראות (ואופציונלית סקריפטים) שסוכן יכול לטעון למשימה. כל כישור הוא ספרייה שמכילה `SKILL.md` (frontmatter + body) ועזרי תוכן נלווים. Cabinet מארח כישורים מקומית, מצמיד אותם לפרסונות, ומרכיב אותם לתוך ספריית העבודה של הסוכן בזמן ריצה.

הדף הזה מכסה מה כישורים יכולים לעשות, איפה הם חיים, ובעיקר — מודל האבטחה שאתם מסכימים לו כשאתם מתקינים אחד.

## מה זה כישור

כל כישור הוא תיקייה, למשל:

```
.agents/skills/imagegen/
├── SKILL.md          # frontmatter (name, description, allowed-tools) + body
├── scripts/          # סקריפטים אופציונליים שהסוכן עשוי להריץ
└── assets/           # קבצי עזר אופציונליים (לוגואים, סכמות וכו׳)
```

ה-`SKILL.md` הוא החלק שהמודל קורא. ה-frontmatter מצהיר על `name`, `description`, ו-(אופציונלית) `allowed-tools`. הגוף הוא ה-playbook שהמודל ילך לפיו כשהכישור רלוונטי למשימה.

## איפה כישורים חיים

Cabinet מסתכל בחמישה מקורות, לפי סדר עדיפות זה בהתנגשות מפתחות:

1. **scoped לארונית** — `data/<cabinet>/.agents/skills/` (גלוי רק בתוך הארונית הזו)
2. **שורש ארונית** — `.agents/skills/` בשורש הפרויקט (המקום הקנוני לכישורים שמתקינים דרך הגדרות → כישורים)
3. **repo מקושר** — כישורים שנמצאים ב-repos מקושרים דרך `.repo.yaml` (עוד לא מחובר)
4. **מערכת** — `~/.claude/skills/` ו-`~/.agents/skills/` (גלובלי לכל המארח, משותף בין CLIs); וגם plugin marketplaces של Claude Code תחת `~/.claude/plugins/marketplaces/`
5. **בית-ישן** — `~/.cabinet/skills/` (נשמר לתאימות אחורה)

המקורות המנוהלים על-ידי Cabinet (1 ו-2) ניתנים לעריכה מה-UI; השאר לקריאה בלבד.

## התקנת כישור

הגדרות → כישורים → **הוסף כישור** מקבל:

- `github:owner/repo` — ה-repo עצמו הוא הכישור
- `github:owner/repo/<skill>` או `github:owner/repo@<skill>` — הכישור הוא תת-ספרייה
- `github:owner/repo#<ref>` — הצמדה ל-branch / tag / commit SHA
- `https://github.com/owner/repo[/tree/<ref>[/path]]` — הדבקת URL של GitHub
- `https://skills.sh/owner/repo[/skill]` — מ-skills.sh
- `gitlab:owner/repo[/<skill>][#<ref>]`
- `local:/absolute/path` — העתקה מספרייה מקומית (נדיר; שימושי לפיתוח)
- `npx skills add <url> --skill <name>` — הדבקת פקודת התקנה כפי שהיא מ-skills.sh

מקור הכישור נכתב ל-`skills-lock.json` (לא ב-git, פר-מכונה) כדי שתוכלו לאחזר שוב או לבדוק חבילה מותקנת. גם SHA-256 נשמר לכל קובץ כדי שזיהוי שינויים יקלוט עריכות ידניות.

## הצמדת כישורים לסוכן

לכל פרסונה יש רשימת `skills:` ב-frontmatter שלה. הגדרות → כישורים מאפשרת להצמיד/לנתק כישורים לכל סוכן, ודף פרטי הסוכן מציג ״מומלצים לתפקיד הזה״ עבור פרסונות עם `recommendedSkills` בתבנית.

אפשר גם לאזכר כישור עם `@` בכל composer כדי לצרף אותו **לריצה הנוכחית בלבד** (לא נשמר לפרסונה). הצ׳יפ של האזכור מופיע עם אייקון ניצוצות סגול, והכישור נטען לצד הכישורים המוצמדים של הפרסונה לאותה שיחה.

## איך כישורים מגיעים למודל

כשריצה מתחילה, Cabinet:

1. פותר כל מפתח כישור נדרש מול הקטלוג (דרך `resolveDesiredSkills`).
2. בונה **אינדקס כישורים** — בלוק שם + תיאור — ומזריק אותו ל-prompt של הסוכן כך שהמודל יודע מה זמין בלי שנטען מראש גוף כישור.
3. מממש tmpdir של plugin פר-סשן ב-`${TMPDIR}/cabinet-skills/<sessionId>/` שמכיל `.claude-plugin/plugin.json` ו-`skills/<key>/` symlinks לכל חבילה.
4. מפעיל את ה-CLI עם `--plugin-dir <tmpdir> --add-dir <tmpdir>`.
5. ביציאת סשן, מוחק את ה-tmpdir.

לתורי המשך על **אדפטרים מובנים** (למשל Claude `claude-local`), כל תור הוא spawn חדש, כך שהטעינה נבנית מחדש. אזכורי `@` באמצע שיחה נקלטים חי.

ל-**continuations במצב טרמינל/PTY**, ה-spawn ארוך-חיים ולא יכול לרשום כישורים חדשים דינמית. אזכורים באמצע סשן עדיין מגיעים למודל דרך טקסט (האינדקס נכתב מחדש בכל תור), אבל הם לא ניתנים לגילוי כפקודות slash חיות עד המשימה הטרייה הבאה.

## מודל אבטחה — קראו את זה לפני התקנת כישורי צד-שלישי

מודל האיום של Cabinet הוא **מקומי בלבד** — ה-daemon מאזין ל-`127.0.0.1`, בלי אימות, משתמש יחיד. בגבול הזה, הבחירות הבאות מכוונות:

### 1. אין שער אמון בזמן ריצה

ברגע שמתקינים כישור ומצמידים אותו לפרסונה, כל ריצה טוענת אותו. אין בקשת הרשאה לכל תור. סיגנלי האמון (מקור, פיל ״עברו audits״, מצאי קבצים, הצהרת `allowed-tools`) חיים ב-installer — ברגע שהותקן והוצמד, הכישור נחשב מורשה לפי המעשה שלכם.

לביטול כישור, נתקו אותו מהפרסונה או מחקו את החבילה מ-`.agents/skills/`.

### 2. Claude רץ עם `--dangerously-skip-permissions`

האדפטר `claude-local` מעביר `--dangerously-skip-permissions` כדי ש-Cabinet יוכל לשדר פלט בלי בקשות הרשאה אינטראקטיביות. זה אומר ש-**כל הוראות בחבילת כישור יכולות להגיד לסוכן להריץ פקודות shell, ו-Claude יבצע בלי לשאול**. אל תתקינו כישור שלא הייתם מוכנים להריץ ממנו סקריפט shell.

### 3. חבילות כישור מקבלות את ה-`.cabinet.env` שלכם

ה-CLIs ש-Cabinet משיק (וכל סקריפט שכישור מפעיל) יורשים את סביבת התהליך של Cabinet, שכוללת כל מפתח ב-`.cabinet.env` — `OPENAI_API_KEY`, `GITHUB_TOKEN` וכו׳. כישור עוין יכול לקרוא אותם. הקובץ הוא `0600` ולא ב-git, אבל החשיפה בתוך התהליך אמיתית.

זה מכוון (זה איך כישורים כמו `imagegen` מקבלים את המפתח שלהם בלי תקלת רץ פר-ריצה), אבל כדאי לדעת. אם צריך לתחום מפתח לכישור אחד בלבד, הגדירו אותו בסביבת ה-shell במקום ו-Cabinet לא יכתוב אותו לדיסק.

### 4. פילי ״audits passed״ הם סיגנלים, לא שערים

הקטלוג מביא תקצירי audit מ-`add-skill.vercel.sh/audit` (Alibaba Threat Hunter, Socket, Snyk, zeroleaks). ספירת מעברים גבוהה היא סיגנל שימושי אבל לא חוסם — אפשר להתקין כישור עם אפס audits שעברו אם רוצים.

### 5. סימלינקים בחבילות נשמרים verbatim

ביבוא חבילת כישור, Cabinet משתמש ב-`fs.cp(..., { verbatimSymlinks: true })` — סימלינקים מועתקים כמו שהם ולא נסרקים. זה מונע מחבילה עוינת לחלץ קבצים דרך סימלינק שמצביע מחוץ לספרייה שלה.

### 6. הגנות מפני path traversal

- מפתחות כישור חייבים להיות kebab-case (`^[a-z0-9][a-z0-9-]*$`, עד 64 תווים). כל דבר שמכיל `/`, `..`, או רווח לבן נדחה.
- scopes של `cabinet:<path>` עוברים דרך `resolveContentPath`, שאוכף את הגבול של `DATA_DIR`.
- מניפסטי plugin שמצהירים על נתיבים יחסיים שיוצאים מהשורש נדחים ביבוא.
- `git clone --branch <ref>` דוחה כל ref שמתחיל ב-`-` או מכיל תווים מחוץ ל-`[A-Za-z0-9._/-]`.

### 7. המקור `local:` לא מוגבל

`local:/absolute/path` מאפשר להעתיק מכל מקום בדיסק אל `.agents/skills/`. זה בסדר ל-daemon מקומי בלבד (כבר אפשר לקרוא את הקבצים האלה), אבל זה אומר שבאג זדוני ב-frontend יכול לחלץ ספריות שרירותיות אל שורש הכישורים. השתמשו בזה לכישורים שאתם מפתחים, לא להתקנות כלליות.

## כתיבת כישור משלכם

צרו ספרייה תחת `.agents/skills/<your-skill-key>/` והכניסו `SKILL.md`:

```markdown
---
name: my-skill
description: תיאור משפט אחד שמשמש באינדקס שמוזרק ל-prompt של הסוכן.
allowed-tools: bash, read, write
---

# הכישור שלי

גוף ה-markdown — ה-playbook שהמודל הולך לפיו כשהכישור הזה רלוונטי.

## מתי להשתמש

- טריגר קונקרטי 1
- טריגר קונקרטי 2

## איך ליישם

צעדים שהסוכן צריך לעקוב אחריהם…
```

אחרי היצירה, רעננו הגדרות → כישורים; הוא יופיע בקטע השורש של Cabinet. הצמידו לסוכן והוא יהיה זמין בריצה הבאה.

לשיתוף, push לתיקייה ל-git repo ואחרים יוכלו להתקין דרך `github:owner/repo/<your-skill-key>`.

## גיבויים

הגדרות → עדכונים → **צור גיבוי** עם שני צ׳ק-בוקסים שמשפיעים על כישורים:

- **כישורים (`.agents/skills/`)** — לכלול חבילות כישור מותקנות בגיבוי. כבוי כברירת מחדל כי הם מצב פר-מכונה.
- **מפתחות API (`.cabinet.env`)** — לכלול את קובץ הסביבה. כבוי כברירת מחדל כי הגיבוי הוא טקסט.

שניהם כבויים כברירת מחדל כך שגיבוי שגרתי לא תופס בטעות סודות או מצב פר-מכונה.

## פתרון בעיות

- **כישור הותקן אבל הסוכן לא משתמש בו** — בדקו את רשימת `skills:` של הסוכן בקובץ הפרסונה. התקנה רק שמה את החבילה בדיסק; הצמדה היא צעד נפרד.
- **שגיאות `--plugin-dir` ב-spawn** — מבנה ה-tmpdir נבנה מחדש בכל ריצה. אם רואים שגיאות ישנות, אתחלו את ה-daemon כדי לנקות את `${TMPDIR}/cabinet-skills/`.
- **אזכור באמצע שיחה לא נרשם במשימת טרמינל** — מגבלה ידועה (ראו ״איך כישורים מגיעים למודל״). שלחו את אותו אזכור ממשימה טרייה והוא ייטען.
- **`skills-lock.json` מציג drift** — Cabinet רושם SHA-256 פר-קובץ בהתקנה. עריכה ידנית של חבילה מדווחת כ-`modified`; התקנה מחדש או הרצת `npm run skills:sync` מסנכרנים את ה-lock.
