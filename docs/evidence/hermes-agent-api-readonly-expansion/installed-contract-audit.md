# Hermes Agent 0.19 read-only contract audit

Audit date: 2026-07-20. Installed runtime identity: Hermes Agent `0.19.0`. Installed checkout: `d7b36070ef807841699ad32c5b6af547fee3ff64` (`v2026.7.20-28-gd7b36070e`). Source: `~/.hermes/hermes-agent/gateway/platforms/api_server.py` (owner-home identity redacted).

This audit used the existing API-key-authenticated loopback server. No Management or Gateway credential was configured, no service was restarted, and no Hermes mutation was performed. The live `/openapi.json` route returned HTTP 404; the installed server is `aiohttp`, and its authenticated `GET /v1/capabilities` response plus the installed source are the exact contract authorities.

| Interface | Installed contract | Authentication and failure boundary | Current/history semantics | Sensitive fields and Cabinet decision |
| --- | --- | --- | --- | --- |
| `GET /health/detailed` | Object with runtime status/version plus gateway and process fields | Bearer API key; Cabinet timeout; redirects refused | Current point-in-time health | Cabinet retains only bounded runtime identity and explicit state. | `GET /v1/capabilities` | Capability object describing auth, endpoints, features, model, platform, and runtime | Bearer API key; source-specific failure; redirects refused | Current installed server contract | Arbitrary nested metadata is not returned to the browser. No durable global run enumeration, Agent-reported active-profile field in the observed live health response, Agent artifact index, or independent usage-history endpoint was found. The configured `operator-os` profile therefore remains configured context, not an observed Agent fact. Session usage fields are shown only when explicitly present on each returned session.

The installed source refuses API-server startup without an API key. All implemented reads use server-side bearer API-key authentication, `cache: no-store`, an abort timeout, and `redirect: error`. No credential-bearing header or raw response enters generated evidence or the browser model.
